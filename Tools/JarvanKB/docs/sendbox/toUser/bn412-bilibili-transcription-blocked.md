# 现在卡在哪:B站视频"转文字"那一步被 B站反爬挡住了

## 一句话
B站 Skill 和 Watcher 的**代码都做完、测过了**。唯一还没在"真实视频"上跑通的,是**把视频下载下来转成文字**这一步——
因为我们用的转写引擎 **BiliNote** 在去 B站下载视频时,被 **B站的反爬(风控)** 拦了,报 `HTTP 412`。

## 打个比方
BiliNote 内部有个"下载器"(叫 yt-dlp)替我们去 B站把视频取回来。B站最近收紧了反爬,看这个下载器**像机器人**,
就拒绝了。`HTTP 412`("前提条件不满足")翻译成人话就是:**"我不信任你这个来访者,不给。"**
**这跟我们写的代码无关**——我们的代码只是把视频链接交给 BiliNote 就完事了,被拦的是 BiliNote 这一步。

## 哪些其实是好的(不用担心)
- **B站 Skill(SP-4b)**:链接 → 调引擎 → 存成 Markdown、还有"模糊路径自动分类"——都通过了,分类还在你真实的
  `mimo` 大模型上实跑成功(把样例视频归到了"机器学习"文件夹)。✅
- **B站 Watcher(SP-5b)**:收藏夹轮询、按"收藏时间(`fav_time`)"只抓新的、cookie 拉取解密、去重、出错时优雅降级——
  **全部对着你的真实账号验证过**。关键是:撞到 412 时它**不崩、不漏、不重复**——这个视频本轮先跳过,**BiliNote 一旦恢复,
  下一轮自动补抓落盘**。✅
- 所以唯一没当面演示的,就是"一个真实视频 → 一篇 Markdown 笔记"这**最后一跳**。

## 为什么"把 cookie 推给 BiliNote"没解决
我们试过把你的登录 cookie 推给 BiliNote 的下载器,**还是 412**。说明 B站拦的**不是"你没登录"**,而是认出了下载器的
**"网络指纹"**(从哪个服务器 IP 来、请求长什么样)。所以光给登录态不够。

## 怎么修(都在 BiliNote / 服务器那一侧,不用改我们的 app)
按"最可能见效 → 最麻烦"排:
1. **换 BiliNote 的出口网络 / IP**(最可能直接好):B站很可能把你这台服务器的 IP 标记了。让 BiliNote 走一个干净的网络出口
   (换机器 / 换网络 / 挂个干净代理)往往一换就通。
2. **升级 BiliNote 里的下载器(yt-dlp)**:B站反爬升级后,yt-dlp 社区通常很快出新版绕过。进 BiliNote 容器把 yt-dlp 更到最新。
3. **给下载器补"伪装头"**(门槛最高):让 BiliNote 的下载器带上更像真浏览器的签名(wbi 等)——这要 BiliNote 本身支持。

> 旁证:同一个引擎 **2026-06-02 是跑通过的**(那次 smoke 成功转写了两个视频)。所以不是引擎坏了,是**这阵子 B站风控收紧了**。

## 还有一个更小的、已经有解的部署提示
启动时如果报 `ImportError: ... socksio`,是因为引擎连本地 BiliNote 时**误用了你系统里的 SOCKS 代理**。
解决:运行环境里**清掉 `ALL_PROXY` / `HTTP(S)_PROXY`** 这几个环境变量即可(已写进 runbook)。这个不影响功能,只是部署时注意一下。

## 你现在的选择
- **(A) 先挂着**:把 412 当一个独立运维项(已记为 **UN-035**)放着。代码已完成,哪天 BiliNote 那侧修好,Watcher 会自动补抓,
  整个 B站垂直照常收敛。**不阻塞别的事。**
- **(B) 现在就修 BiliNote**:想现在通,最快是上面第 1 条(换出口 IP)。**要的话我一步步指引你**(进容器、改 docker 网络/出口、
  或升级 yt-dlp 的具体命令)。
