## 1. What v1.1 adds (scope, user-confirmed 2026-06-07/09)

Five cohesive changes, all inside the watcher service's config / favorites_client / watcher / state layers:

1. **Target resolution modes** — config expresses *what to watch* as a unified `targets` list:
   `type: collection` (explicit, ≈ v1) and `type: user` (auto-discover a user's **named** collections; the
   default `我的收藏` is **skipped by default** — it waits for the classifier SP, §9/§10).
2. **Watermark correction** — parse the now-confirmed **top-level `created`** (favorite time) per item; add an
   optional `only_after` filter + a **first-run baseline** so a fresh by-user watch does not back-fill years
   of history. **No early-stop** (empirically unsafe — items are not favorite-time-ordered).
3. **403 backoff (lightweight)** — a failure-counter/cooldown so permanently-403 items (e.g. 专栏 articles)
   are not re-attempted every cycle.
4. **Forward-compat hooks** — preserve `is_default` (collection) + the per-item excerpt fields so the future
   default-collection-classifier SP needs no re-fetch (that classifier is a SEPARATE SP — §10).
5. **Doc corrections + Step-8 promotions** — fix v1 design §2's false "no favorited-time" claim; record the
   real field semantics in `crawl-pipeline.md §知乎链路`; promote the empirical-first methodology globally.

**Explicitly NOT in v1.1**: the Engine ARTICLE-403 fallback — already investigated **and declined** by SP-2
v1.2 (专栏 `/api/v4/articles/{id}` is `x-zse-96`-gated; the user chose **no signer**, UN-028), so 专栏 full
content stays **permanently unfetchable** via the engine (§7/§10); and the default-collection auto-classifier
— a separate SP (handoff written, §10).

## 2. Empirical basis (corrections to v1's assumptions)

Full detail + provenance: the fields doc. The three findings that shape this design:

- **`/api/v4/me` works unsigned (HTTP 200, plain cookie)** → resolving `url_token: me` is safe.
- **Each collection item has a top-level `created`** (ISO8601, sibling to `content`) = the **favorite time**.
  v1 read `content.created` (authoring time) and wrongly concluded no favorite-time field exists.
- **Items are NOT ordered by favorite time** (observed: a collection's item[0] favorited 2025-08, item[1..2]
  favorited 2026-05). So a high-watermark *filter* is safe but an *early-stop* is **not** — it would skip
  newer items buried behind older ones. (The exact sort key was not determined; a real early-stop would need
  a follow-up probe for an `order_by`/`sort` param — deferred, user chose not to pursue it now.)
- **`is_default: true`** on `/people/{token}/collections` cleanly flags `我的收藏`; the list is `updated_time`
  DESC; each collection carries `item_count`.

## 3. Config schema — unified `targets` (clean replace of v1 `collections`)

```yaml
poll_interval_minutes: 45
# the existing Zhihu library (built by Zhihu-Collection-Saver); ~33 category subfolders already present,
# so named-collection targets land in their matching existing subfolder (continue the library)
output_dir: "/home/shenzhou/Documents/Obsidian/JasonJarvan/ResourceBase资源库/Zhihu知乎"
state_dir: /data/state

cookie_source: { base_url: ..., uuid: ..., password: ... }   # unchanged from v1

# NEW: optional global watermark guards (see §6)
only_after: "2026-01-01T00:00:00+08:00"   # optional; skip items favorited on/before this
backfill_on_first_run: false              # default false: first sight of a collection sets a baseline

# NEW: optional backoff tuning (see §7)
max_consecutive_failures: 3               # default 3
failure_cooldown_hours: 24                # default 24

targets:
  - type: collection            # Mode 2 — explicit (≈ v1; `id` may be id or full URL)
    id: "630144608"
    name: "AI-papers"           # optional; defaults to the collection id
  - type: user                  # Mode 1 — auto-discover a user's NAMED collections
    url_token: me               # "me" (resolved via /api/v4/me) or a literal token
    name_prefix: "zhao/"        # optional subfolder prefix for this user's collections
    skip_empty: true            # optional; default true — skip item_count==0 collections
    include_default: false      # optional; default false — SKIP 我的收藏 (is_default); it waits for the
                                # classifier SP (§9/§10). Set true only to dump it flat into 我的收藏/.
```

- **Clean replace**: v1's top-level `collections:` key is **removed** (no back-compat alias — v1 is freshly
  merged with only an example config; the loader raises a clear error if `targets` is missing).
- **Validation**: ≥1 target; `collection` requires `id`; `user` requires `url_token`.
- Dataclasses: `CollectionTarget(id, name)`, `UserTarget(url_token, name_prefix, skip_empty, include_default)`,
  `Target = CollectionTarget | UserTarget`; `WatcherConfig.targets: list[Target]` (+ the new scalar fields).

## 5. Data flow (updated `run_cycle`)

```
run_cycle():
  cookies = cookie_provider.get_cookies()                 # unchanged
  collections = resolver.resolve(cfg.targets, cookies)    # NEW: per-cycle (picks up newly-created colls;
                                                          #      a user-target 403 degrades, never crashes)
  for coll in collections:                                # existing per-collection loop, with:
      baseline = watermark_store.get_or_init_baseline(coll.id, items, cfg)   # first-run floor (§6)
      seen     = watermark_store.load(coll.id)
      for item in favorites_client.list_items(coll.id, cookies):   # list_items now parses favorited_at + excerpt
          if item.favorited_at and item.favorited_at <= floor: continue      # watermark filter (no early-stop)
          if item.key in seen: continue                                       # dedup (v1)
          if failure_store.should_skip(coll.id, item.key, now): continue      # backoff cooldown (§7)
          doc = fetch(item.url, cookies)
          if doc is None: failure_store.record_failure(...); continue         # fetch failed
          save(...); seen.add(item.key); store.save(...); failure_store.clear(...)
```

`floor = max(only_after, baseline)` where defined. Resolution + filtering are additive in front of the
**frozen v1 fetch/dedup/save** path.

## 6. Watermark correction (user-confirmed strategy)

- **seen-id set stays the correctness backbone** (robust regardless of ordering; already shipped + proven).
- `list_items` **parses the top-level `created` → `CollectionItem.favorited_at`** (timezone-aware datetime).
- **`only_after`** (optional config): skip items with `favorited_at <= only_after`. A pure filter — the loop
  still pages the whole collection (no early-stop).
- **First-run baseline** (`backfill_on_first_run: false`, the default): the first time a collection is polled
  with no prior state, record `baseline = now` (wall clock at first sight), persisted per collection. Items
  favorited at/before the baseline are skipped → only favorites added *after* deployment are saved. Setting
  `backfill_on_first_run: true` disables the baseline (saves all existing history, subject to `only_after`).
- **No early-stop** — §2: items are not favorite-time-ordered, so early-stop would drop newer-but-later-listed
  items. The §知乎链路 paging stop rule (`is_end` / `len >= totals`, never `offset >= totals`) is unchanged.

> Rationale recorded so a future reader does not "re-optimize" into an unsafe early-stop.

## 7. 403 backoff (lightweight, time-window)

- `failure_store` is **separate from the seen-set** (failed items are not "seen"). Key = the same `type:id`.
- On a fetch/save failure: `record_failure` increments the consecutive count; when it reaches
  `max_consecutive_failures` (default 3), set `skip_until = now + failure_cooldown_hours*3600`.
- While `now < skip_until`, the item is skipped (no fetch) — bounds the waste (e.g. the 28 专栏 articles).
- After the cooldown window it is retried once; if it fails again it re-enters cooldown (flat, not
  exponential). **Any success → `clear`** (the count + skip_until are wiped; a 403 may be a transient cookie
  lapse). Time-window (not cycle-count) so it survives restarts / interval changes; `now_fn` injected for tests.
- 专栏 articles 403 **permanently** (SP-2 v1.2 declined a signer — §1/§10), so they never succeed: the
  cooldown bounds them to ~one retry per `failure_cooldown_hours` instead of every cycle. Their list
  **excerpt** is still available (the basis for the future classifier, §9/§10).

## 8. `me` resolution

`get_current_url_token(cookies)` → `GET /api/v4/me` (plain cookie, `trust_env=False`) → `url_token`.
Empirically 200/unsigned (§2), so no residual blocker. A literal `url_token` skips this call. (No caching in
v1.1 — one `/me` call per cycle per `me`-target is negligible; caching is a deferred optimization.)

## 9. Forward-compat hooks (for the future classifier SP — build now, near-zero cost)

`UserCollection` preserves **`is_default`** + `item_count`; `CollectionItem` preserves the per-item excerpt
(`excerpt_title` for articles / `excerpt` for answers → a single `excerpt` field) + resolved `title`. v1.1
does **not** classify — these fields are merely parsed-and-carried so the separate classifier SP can target
the default collection and classify from the list excerpt without re-fetching. (Classifier rationale + the
shared-`(title, lead_text)` design are in the handoff, §10.)

The **classifier's category vocabulary already exists**: `output_dir` (the Obsidian Zhihu library) holds ~33
curated subfolders (技术-*/生涯-*/娱乐-*/知识-*/赚钱-* + a flat `我的收藏`). That is exactly what SP-3's
`classify.existing_subfolders(output_root)` reads — so the future classifier classifies the default-inbox
items into these existing folders, no new taxonomy needed. (In v1.1 the default `我的收藏` is **skipped** by
Mode-1 — `include_default: false`, user-decided — so it is left untouched for the classifier SP. Setting
`include_default: true` instead dumps it flat into the existing `我的收藏/`.)

## 10. Out of scope (re-triaged) + handoffs written

- **Engine ARTICLE-403 fallback** — NOT pursued. Already handled by the orchestration layer: SP-2 v1.2
  dispatched a probe, found 专栏 `/api/v4/articles/{id}` is `x-zse-96`-gated (mirroring the ANSWER fallback is
  infeasible), escalated to root (UN-028), and the **user chose no signer** → SP-2 v1.2 re-scoped to
  rate-limit hardening. ⇒ 专栏 full content is permanently unfetchable via the engine; v1.1's backoff (§7)
  bounds the retry waste, and the list excerpt (§9) remains the only 专栏 content path. (My earlier
  `from-zhihuwatcherv11impler-engine-article-403-gap.md` finding was superseded by this and is obsolete.)
- **Default-collection auto-classifier** — new SP; reopens SP-5a's no-LLM boundary; needs a shared classifier
  extracted from SP-3; classifies into `output_dir`'s existing ~33 subfolders → handoff
  `docs/sendbox/toZhihuCrawlOrche/from-zhihuwatcherv11impler-default-collection-classify.md`.
  *Decided:* v1.1 Mode-1 **skips** the default `我的收藏` (`include_default: false`) so the classifier SP owns
  it — no flat-then-reclassify churn. (`include_default: true` opts into a flat dump if ever wanted.)
- Deferred (future, not handed off): a general collection allow/deny **filter** beyond `skip_empty`;
  **exponential** backoff; `me`-result **caching**; the early-stop **sort-param probe**; the `collections`
  back-compat **alias**.

## 11. Testing strategy (TDD additions; v1's 32 tests stay green)

- `favorites_client`: `list_user_collections` paging (mock; assert paging-to-end, no signing header,
  `trust_env=False`, parses `id/title/is_default/item_count`); `get_current_url_token` parses `url_token`;
  `CollectionItem.favorited_at` parsed from top-level `created`; excerpt resolved per type (article vs answer).
- `resolver`: collection pass-through; user expansion; `me` resolution; dedup-by-id (explicit name wins);
  single-user-target failure degrades (explicit targets still resolved); `skip_empty`; **skips `is_default`
  by default, includes it when `include_default: true`**.
- `failure_store`: count increments → cooldown after N; `should_skip` honors `skip_until` (fake clock);
  success clears; reload persistence; atomic write.
- `watermark`/baseline: first-run baseline set; `only_after` + baseline filter `favorited_at`; `backfill=true`
  disables baseline.
- Integration: full cycle with a `user` target (mock `/me` + `/people/.../collections` + items + SP-2 fetch)
  → correct subfolders + seen advanced; **second cycle = no-op** (dedup invariant preserved); a 403 item
  enters cooldown and is skipped next cycle; first-run baseline skips pre-existing items.
- **Live smoke (verification gate)**: real SP-1 + the real account — `me` resolves; by-user discovers the
  user's **named** collections (default `我的收藏` skipped, empties skipped); an explicit + a by-user target
  both poll; baseline skips history on first run; re-run = no-op.
