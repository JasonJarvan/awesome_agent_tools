## 1. Background & the no-LLM boundary amendment (add-only)

v1 of the watcher locked a **"No LLM / no classification"** boundary (v1 design §1: *"Unlike SP-3, the watcher
does not use LLMClient and does not auto-classify."*). v1.2 **explicitly reopens** that boundary — not by
stealth — under the following amendment, recorded here and in module `decisions.md`:

> **Amendment (2026-06-14, SP-5a v1.2):** classification becomes a **per-target opt-in flag, default OFF**.
> When no target sets `classify: true`, the watcher's behavior is byte-for-byte the v1.1 no-LLM daemon —
> `LLMClient` is never constructed, no network call to any LLM provider is made. The "no-LLM default" is
> therefore **preserved**; the LLM runs only when a target explicitly enables `classify`. This is an
> additive capability behind an off-by-default switch, consistent with the CLAUDE.md §4 *add-only* invariant.

**Primary use:** the default collection **`我的收藏`** (id `721323262`, `is_default=true`) — a flat inbox
the v1.1 watcher deliberately skips (`include_default: false`). v1.2 lets it be watched **and** auto-sorted
into the user's existing ~33 Obsidian category subfolders, token-frugally. **Named collections stay
`classify: false`** (they are already user-categorized by folder name).

**Unchanged upstream contracts** (pure consumer, never edited): SP-2 Zhihu engine `fetch()`, SP-1 cookie
active-pull, the v1.1 target resolver / watermark floor / seen-set.

## 2. Per-item flow when `classify: true` for a target

```
list item (favorites API; carries title + excerpt + favorited_at, even on later 403)
   │  dedup: item.key in LEDGER?  ──yes──► skip
   │  favorited_at <= floor (only_after / first-run baseline)?  ──yes──► skip
   │  failure_store.should_skip / circuit-broken?  ──yes──► skip
   ▼
fetch full content via frozen SP-2 engine (the primary path — 专栏 full text IS fetchable with fresh __zse_ck)
   │  fetch fails (None / 403 / exception)  ──► record_failure ──► (NO save, NO classify, NO ledger) ──► retry next cycle
   ▼
tiered classify on the ALREADY-FETCHED content (no extra fetch):
   Tier-1: lead_text(content_markdown, 200) → classify(..., allow_new=False) → (folder, vague)
   if vague:  Tier-2: lead_text(content_markdown, 1000) → classify(...) → (folder, vague)   [NEVER full text]
   if folder ∉ existing_subfolders (slug-matched):  raise ClassifyError ──► record_failure ──► retry
   ▼
save: <output_dir>/<classified_folder>/<sanitize(title)>.md   (reuse v1.1 saver verbatim; folder = classified)
   ▼
ledger.record(item.key, classified_folder, local_path, fav_time, classified_at, sync_status="local-only")
   ▼
failure_store.clear(item.key)   (success resets any failure record)
```

An item is **"done" (recorded in the ledger) only when fetch + classify-to-an-existing-folder + save all
succeed.** Any failure leaves it un-ledgered → it is retried next cycle (subject to backoff/circuit-break §5).

**403 degrade is RETRY, not excerpt-classify.** A failed fetch (the typical 专栏 nav-GET 403 from a stale
`__zse_ck`) is **not** classified from the list excerpt and **not** stub-saved. It is recorded as a failure
and retried; a fresh `__zse_ck` (synced from the browser via cookie-manager) lets the next cycle fetch the
full content. (This supersedes the handoff §1.3 "classify from list-excerpt" degrade option — user decision
2026-06-14: failures retry + circuit-break rather than persist a degraded artifact.) The list excerpt is
used **only** for the human attention surface (§5), never as a classification input.

## 3. Shared classifier — extracted into `jarvankb_common`

### 3.1 New module `jarvankb_common/classify.py` (self-contained, public API)

```python
@dataclass
class Classification:
    folder: str      # slugified chosen folder name
    is_new: bool     # ALWAYS filesystem-authoritative: slug(folder) ∉ existing_subfolders
    vague: bool      # model-reported: ≥2 folders are plausible (drives the watcher's Tier-1→Tier-2 escalation)

def lead_text(markdown: str, max_chars: int) -> str: ...          # markdown-noise stripper + cap (moved from SP-3)
def existing_subfolders(output_root: Path) -> list[str]: ...      # sorted immediate non-hidden subdirs (moved from SP-3)

def classify(title: str, lead_text: str, existing_subfolders: list[str], client,
             *, allow_new: bool = True) -> Classification: ...
```

- **Self-contained:** owns private `_slugify`, `_parse` (prose-tolerant JSON, prefers last object), and the
  prompt builder. It does **not** import either skill's `saver.slugify` — the shared `_slugify` is for the
  classifier's own folder-name safety. (SP-3 and bilibili-crawl keep their own `saver.slugify` for filenames;
  the ~3-line duplication is accepted, consistent with the SP-5a v1 "minor duplication accepted" decision.)
- **New `vague` output:** the prompt instructs the model to set `vague=true` when two or more existing
  folders are plausible. SP-3 ignores it (single-shot); the watcher acts on it (Tier escalation).
- **New `allow_new` flag:** changes **only the prompt** — when `False`, it instructs *"pick ONLY from the
  existing list; do not invent a new folder."* It does **not** mutate `is_new`. The classifier does not
  itself guarantee an in-list result (the model may still hallucinate off-list); `is_new` truthfully reports
  whether it did, and the **caller** decides what that means (the watcher treats `is_new=True` under
  `allow_new=False` as off-list → `ClassifyError`, §4).
- **`is_new` semantics:** ALWAYS filesystem-authoritative — `slug(folder) ∉ existing_subfolders`, never the
  model's self-reported flag, regardless of `allow_new`.
- Exposed via `jarvankb_common/__init__.py`: `classify`, `Classification`, `lead_text`, `existing_subfolders`
  (alongside the existing `LLMClient`).

### 3.2 SP-3 becomes a thin adapter (non-breaking)

`Skill/crawl/zhihu-crawl/src/zhihu_crawl/classify.py` shrinks to a delegation shim:

```python
from jarvankb_common.classify import classify as _classify, lead_text, existing_subfolders  # noqa: F401

@dataclass
class Category:           # preserved return shape so api.py is untouched
    name: str
    is_new: bool

def classify(result, output_root, client, *, snippet_chars: int = 240) -> Category:
    subs = existing_subfolders(output_root)
    lead = lead_text(result.content_markdown, snippet_chars)
    c = _classify(result.title, lead, subs, client, allow_new=True)   # SP-3 keeps its propose-new behavior
    return Category(name=c.folder, is_new=c.is_new)
```

- `zhihu_crawl/api.py` is **unchanged** (`cat.name`, `cat.is_new` still work).
- `zhihu_crawl/saver.py` is **unchanged** (keeps its own `slugify` for filenames).
- **Behavior delta accepted by the handoff:** the shared prompt is generic `(title, lead)` and drops SP-3's
  explicit `Item type:` line. The handoff blesses this ("SP-3 supplies it from its `content_markdown` lead —
  its current behavior, richer-or-equal"). The lead carries the topic signal; folder-bucket accuracy is
  unaffected in practice.
- **Test migration:** SP-3's `test_classify.py` cases for `_build_prompt`/`_parse`/`_lead_text`/
  `existing_subfolders` move to `jarvankb_common`'s test suite. SP-3 keeps a delegation test (adapter maps
  `Classification → Category`, `allow_new=True`). **Full SP-3 suite re-run after the move (regression gate).**

## 4. Watcher classify wrapper (`classifier.py`) — the tiering lives here

The shared classifier is single-shot; tiering is watcher-specific (SP-3 does not tier), so a new
`src/zhihu_watcher/classifier.py`:

```python
class ClassifyError(Exception): ...   # off-list after Tier-2, or unparseable → routed to the failure path

def classify_doc(doc, existing_subfolders, client, cfg) -> str:   # returns the chosen folder name
    lead1 = lead_text(doc.content_markdown, cfg.tier1_chars)      # 200
    c = classify(doc.title, lead1, existing_subfolders, client, allow_new=cfg.allow_new_folders)
    if c.vague:
        lead2 = lead_text(doc.content_markdown, cfg.tier2_chars)  # 1000  (NEVER full text)
        c = classify(doc.title, lead2, existing_subfolders, client, allow_new=cfg.allow_new_folders)
    if not cfg.allow_new_folders and c.is_new:        # model went off-list despite the instruction
        raise ClassifyError(f"off-list folder {c.folder!r} with allow_new=False")
    return c.folder
```

- `allow_new_folders` defaults **False** for the watcher (user decision 2026-06-14): an unattended daemon
  must not fractalize the taxonomy. Off-list-after-Tier-2 (`c.is_new` True) → `ClassifyError` → failure path
  (no junk folder; the item retries and surfaces in attention if it persistently can't be placed). The
  membership test reuses the classifier's filesystem-authoritative `is_new` — the watcher needs no slug logic.
- The `LLMClient` is constructed **once per cycle, lazily, only if a classify-enabled target exists** (so a
  pure-named-collection config never touches the LLM — preserving the §1 no-LLM-default amendment).
- Classification runs on `doc.content_markdown` (the fetched full text); tiers differ only in the lead cap.

## 5. Stores: ledger + failure escalation + attention surface

### 5.1 Classification ledger — new `ledger_store.py` (service-side; engine stays stateless)

One JSON file per classify-enabled collection: `<state_dir>/ledger-{collection_id}.json`, keyed by `item.key`
(`"{content_type}:{content_id}"`):

```json
{
  "answer:12345": {
    "classified_folder": "机器学习",
    "local_path": "/…/Zhihu知乎/机器学习/标题.md",
    "fav_time": "2026-05-23T09:00:04+08:00",
    "classified_at": "2026-06-14T08:30:00+00:00",
    "sync_status": "local-only",
    "sync_attempts": 0
  }
}
```

- **The ledger is the authoritative dedup for classify-enabled targets** — independent of any Zhihu API.
  For a classify target, dedup = `item.key in ledger`; the watcher does **not** also write the v1.1 seen-set
  for that collection (one record, not two). Non-classify targets are unchanged (seen-set only). This is the
  fallback-by-design for v1.3: even if sync is never built, the ledger still prevents re-classification.
- **Records SUCCESSES only** (a failed fetch/classify is never recorded → it retries).
- Persisted immediately after each successful item (atomic tmp+`os.replace`, mirrors `WatermarkStore`).
- `sync_status` / `sync_attempts` are **v1.3 forward-compat**: v1.2 writes the defaults (`"local-only"`, `0`)
  and never reads them. v1.3 (【同步知乎收藏夹分类】) will use them to drive the Zhihu write-API move. **Not
  implemented in v1.2.**

### 5.2 Failure escalation ladder — extend v1.1 `FailureStore`

A single ladder for **all** per-item failures (403 fetch, `ClassifyError`, save error):

| Stage | Trigger | Effect |
|---|---|---|
| retry | failures `< max_consecutive_failures` (default 3) | retried every cycle |
| cooldown | failures `>= max_consecutive_failures` | skipped until `now + failure_cooldown_hours` (default 24) |
| **circuit-break** | failures `>= circuit_break_threshold` (default 10, **new**) | **permanently skipped** + written to the attention surface + `log.error` |

- During retry/cooldown (pre-circuit-break): **no stub file, no classify, no ledger entry.**
- `FailureStore.record_failure` gains optional `url` / `title` / `excerpt` args (the list always has these,
  even for an un-fetchable 专栏) so the attention renderer has something human-readable. `clear()` on success.
- `circuit_break_threshold` is **top-level config** (parallel to `max_consecutive_failures`), because the
  ladder applies to fetch failures too, not only classify failures.

### 5.3 Attention surface — JSON authority + Markdown render (two layers)

- **Authoritative state:** the `FailureStore` JSON in `state_dir` (machine round-trips cleanly).
- **Human surface:** `<output_dir>/_zhihu-watcher-attention.md` — **regenerated by integral overwrite every
  cycle** from the circuit-broken entries across all collections (never parsed back → no markdown round-trip
  fragility). Obsidian renders it natively: clickable URL, title, excerpt, failure count, first/last-seen,
  and a **probable-cause hint** (stale `__zse_ck` cookie vs. permanently-unreachable content). When nothing
  is circuit-broken, it writes a single line: `✓ nothing needs attention`.
- Purpose: when an item burns through the ladder, the user reads this list to judge the **real** cause and
  act (refresh the browser cookie, or accept the item is unreachable).

## 6. Config surface

```yaml
# … v1.1 fields unchanged …
max_consecutive_failures: 3
failure_cooldown_hours: 24
circuit_break_threshold: 10          # NEW: permanent-skip + attention threshold

classify:                            # NEW block; only consulted when a target sets classify: true
  llm_profile: mimo-v2.5-pro         # reuse config/llm.yaml profile (already configured)
  tier1_chars: 200
  tier2_chars: 1000
  allow_new_folders: false           # unattended daemon: pick from existing folders only

targets:
  - type: collection
    id: "721323262"                  # 我的收藏 (default inbox)
    name: "我的收藏"                 # ignored when classify: true (items land in their classified folders)
    classify: true                   # NEW per-target flag (default false)
  - type: user
    url_token: me
    include_default: false           # named collections stay classify=false (default)
```

- `CollectionConfig` gains `classify: bool = False`. The v1.1 `TargetResolver` sets `classify=False` for all
  user-discovered collections (named-collection classify is out of scope for v1.2); explicit `type: collection`
  targets carry their own flag.
- A `ClassifyConfig` dataclass (`llm_profile`, `tier1_chars`, `tier2_chars`, `allow_new_folders`) is parsed
  from the `classify:` block; absent block → classify globally disabled regardless of per-target flags
  (defensive: a `classify: true` target with no `classify:` block is a config error, surfaced at load).

## 7. Error handling & daemon-never-crashes (unchanged invariants)

The v1.1 guards hold: cookie pull, list, state load, and per-item processing are each wrapped so one failure
logs and continues. v1.2 additions:
- `ClassifyError` and any `LLMClient` exception inside `_poll_collection`'s per-item block are caught by the
  existing broad `except` → `record_failure` → continue (the LLM is just another external dependency).
- Attention-surface render failure is caught and logged — it must never abort a cycle.
- A `classify: true` target whose `LLMClient` cannot be constructed (no creds) → the cycle logs and skips
  that target (does not crash; non-classify targets still run).
